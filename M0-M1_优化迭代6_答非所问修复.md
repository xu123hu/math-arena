## 1. 优化概述

### 1.1 背景

在 M0-M1 阶段的用户测试中，我们发现模型存在严重的"答非所问"问题。该问题在"超纲但合理"的数学问题上尤为突出——当用户询问超出教材知识库范围、但模型本身具备解答能力的概念时，模型不仅不回答，反而以"不属于高中知识范畴"为由拒答，与用户的求知需求产生直接冲突。

### 1.2 具体案例

| 轮次 | 用户输入 | 模型实际响应（修复前） | 期望行为 |
|------|---------|----------------------|---------|
| 第 1 轮 | "什么是薛定谔方程" | "薛定谔方程不属于高中知识范畴，我无法回答。" | 基于通用数学能力简要回答，标注超出教材范围 |
| 第 2 轮 | "可是我很想知道" | 罗列高中物理/化学知识点，仍未回答方程本身 | 结合上下文理解追问意图，给出方程的核心含义 |

第 2 轮的问题尤为严重：用户明确表达了强烈的求知意愿，但模型不仅没有回答，反而"答非所问"地罗列知识点，说明**上下文已断裂**——模型丢失了上一轮的对话焦点。

### 1.3 优化目标

本次迭代聚焦修复三个核心问题：

1. **答非所问**：超纲但合理的求知问题被错误拒答，模型应基于通用数学能力给出解答
2. **过度限制**：防幻觉规则将"不在教材知识库中"误判为"概念不存在"，触发保守拒答
3. **上下文断裂**：降级路径绕过 ContextAssembler，导致多轮对话历史和滚动摘要丢失，追问时模型丢失上下文

## 2. 问题诊断与根因分析

经代码审查与实际复现，确认三重根因，三者叠加共同导致了"答非所问"现象。

### 根因 A — 降级路径绕过上下文装配

**文件**：`services/api/app/skills/qa_rag/main.py`（修改前第 147-163 行）

当 RAG 判定 `answerable=False`（知识库未命中相关内容）时，降级路径**手动构建 messages 列表**，仅包含一个 system 消息和一个 user 消息（question），完全绕过了 `ContextAssembler.assemble()`。

```python
# 修复前（伪代码示意）
if rag_result is None or not rag_result.answerable:
    messages = [
        {"role": "system", "content": "...防幻觉规则..."},
        {"role": "user", "content": question},  # ← 仅含裸 question
    ]
```

**后果**：
- `working_memory`（对话历史 + 滚动摘要）未被注入降级上下文
- `user_profile`（用户档案）未被注入
- 当用户追问"可是我很想知道"时，模型看到的只有这 5 个字，**完全丢失了上一轮讨论"薛定谔方程"的上下文**，导致答非所问

### 根因 B — 降级 prompt 防幻觉规则自相矛盾

**文件**：`services/api/app/skills/qa_rag/main.py`（修改前第 152-157 行）

降级路径的 system prompt 中，包含 4 条"严格执行"的防幻觉规则，其核心指令为"如果概念不存在或你不确定，直接告知该概念不存在"。然而紧随其后的第 157 行又写道"可基于通用数学能力简要回答"。

```text
# 修复前 prompt 结构（矛盾示意）
- 严格执行：如果概念不存在或你不确定，直接告知该概念不存在   ← 强拒答
- 严格执行：不要对概念进行假设性解释                       ← 强拒答
- 严格执行：不要编造推测性内容                             ← 强拒答
- 严格执行：回答必须基于知识库内容                         ← 强拒答
- 可基于通用数学能力简要回答                               ← 弱许可（被压制）
```

**后果**：模型在指令冲突时倾向于执行更保守的限制。"薛定谔方程"不在教材知识库中 → 被误判为"概念不存在" → 触发拒答。第 157 行的许可指令被前 4 条"严格执行"完全压制。

### 根因 C — persona 矛盾表述

**文件**：`services/api/app/kernel/prompts/student.md`（修改前）

student.md 中存在多处自相矛盾的表述：

| 位置 | 表述 | 倾向 |
|------|------|------|
| 第 7 行 | "超出知识库范围，诚实说明并给出通用解答" | 许可回答 |
| 第 36 行 | "对于超出范围的问题，诚实说明并给出通用解答" | 许可回答 |
| 第 38-43 行 | 防幻觉规则"如果概念不存在…直接告知不存在" | 强拒答 |
| 第 57 行（修改前） | "不讨论与数学无关的话题" | 强限制 |

**后果**：第 7/36 行的"通用解答"许可与防幻觉规则的"拒答"倾向冲突，模型倾向于执行更保守的限制（根因 B 的同类问题在 persona 层面复现）。此外，修改前第 55-60 行"不讨论与数学无关的话题"表述过于刚性，与"给出通用解答"形成矛盾，模型倾向于"不回答"而非"给出通用解答后说明范围"。

### 根因链路总结

```
用户问"薛定谔方程"
    │
    ▼
RAG 检索 → answerable=False（知识库无此内容）
    │
    ▼ 降级路径
    ├─ 根因A：绕过 ContextAssembler → 上下文断裂（追问丢失历史）
    ├─ 根因B：防幻觉规则压制"通用回答" → 误判"概念不存在" → 拒答
    └─ 根因C：persona 矛盾 → 模型倾向保守限制
    │
    ▼
"不属于高中知识范畴，我无法回答"（答非所问）
```

## 4. 分级响应策略说明

本次修复的核心设计是**分级响应策略**，替代原先"一刀切拒答"的降级逻辑。该策略在 `qa_rag/main.py` 降级路径的 `output_spec` 和手动构建 prompt 中均有体现，并在 `student.md` persona 层面形成一致指导。

### 4.1 三级响应分级

| 级别 | 触发条件 | 响应行为 | 示例 |
|------|---------|---------|------|
| **L1 虚构概念** | 概念/定理/公式在模型知识中确实不存在或无法确认真实性 | 明确告知"我无法确认该概念的真实性"，不编造 | "张氏收敛定理"、"李氏超限递归原理" |
| **L2 超纲但真实** | 概念确实存在，但超出教材知识库范围 | 基于通用数学能力给出准确、简洁的解释，开头标注"以下内容超出当前教材范围，仅供参考" | "薛定谔方程"、"黎曼猜想"、"傅里叶变换" |
| **L3 教材内** | 概念在知识库命中（answerable=True） | 基于 RAG 引用回答，标注【N】（正常路径，本次未改动） | "什么是导数" |

### 4.2 追问感知

当用户追问表达强烈求知意愿时（如"我很想知道"、"你能告诉我吗"），模型应**尊重并给出能确定的解答**，而非继续拒答。这一规则在分级响应 prompt 第 3 条中明确写入：

```text
3. 如果用户追问表达强烈求知意愿（如'我很想知道'、'你能告诉我吗'），
   应尊重并给出你能确定的解答
```

### 4.3 设计原则

分级响应策略遵循以下原则：

1. **区分"真不知道"与"知道但超出教材"**：前者拒答（L1），后者回答（L2），避免将"知识库外"等同于"概念不存在"
2. **尊重求知意愿**：用户明确想学时，模型应积极回应而非设限
3. **标注透明性**：超纲回答必须标注"超出教材范围，仅供参考"，不伪造教材引用
4. **防护层兜底**：`degraded=True` 时 guard.py 自动移除所有【N】引用标记，防止模型在降级路径中伪造教材来源

### 4.4 决策流程图

```
用户提问
    │
    ▼
RAG 检索
    │
    ├── answerable=True ──→ L3：基于引用回答（正常路径）
    │
    └── answerable=False ──→ 降级路径
                                │
                                ▼
                          模型判断概念是否真实存在
                                │
                    ┌───────────┴───────────┐
                    ▼                       ▼
              真实但超纲                虚构/无法确认
              (L2)                     (L1)
              通用能力回答              告知无法确认
              标注超出教材              不编造
```

## 5. 上下文装配改进说明

### 5.1 改动核心

降级路径（`answerable=False`）原先手动构建仅含 `question` 的裸 messages，现在改为调用 `ContextAssembler.assemble()`，注入完整的上下文层级。

### 5.2 注入的上下文层级

| 层级 | 内容 | 来源 | 作用 |
|------|------|------|------|
| P0 System Persona | 角色提示词（student.md/teacher.md） | `_load_persona(active_role)` | 定义模型行为准则 |
| P1 User Message | 当前问题 | `user_message=question` | 保命段，永不裁剪 |
| P4 Working Memory | 对话历史（最近消息）+ 滚动摘要 | `working_memory` | **修复上下文断裂**——追问时模型能看到之前的讨论焦点 |
| P5 User Profile | 用户档案（年级、水平、薄弱点） | `user_profile` | 个性化回答 |
| P7 Output Spec | 分级响应指导 | `output_spec` 参数 | 指导降级路径的响应行为 |

### 5.3 预算与裁剪保护

上下文装配受 **12K token 总预算** 控制，裁剪链为 `P3→P5→P4→P6`：

```
P0 system_persona   (800)  ─┐
P1 user_message    (2000)   ├── 保命段，永不裁剪
P2 skill_params     (600)  ─┘

P3 rag_chunks      (4000)  ← 最高优先裁剪（降级路径无 RAG，不涉及）
P5 user_profile     (500)  ← 第二优先裁剪
P4 working_memory  (1600)  ← 第三优先裁剪（保留最近 3 条 + 摘要最短 50 字）
P6 episodic         (800)  ← 最低优先裁剪

P7 output_spec      (400)  ← 附加到 system core
```

降级路径不传 `rag_chunks`，因此 P3 为空，实际可用预算更充裕，`working_memory` 和 `user_profile` 有更大保留空间。

### 5.4 guard.py 降级模式兜底

`_stream_generate` 在降级路径中标记 `degraded=True`，guard.py 的 `check_output` 方法检测到该标记后，自动移除所有【N】引用标记：

```python
# guard.py 第 114-140 行
async def check_output(self, text, ctx, *, valid_chunk_ids=None, degraded=False):
    # 0. 降级模式：移除所有【N】标记，防止伪造引用
    if degraded:
        text = self._strip_degraded_citations(text)
    # ...

_DEGRADED_CITATION_RE = re.compile(r"【\d+】")

def _strip_degraded_citations(self, text):
    """降级模式下移除所有【N】引用标记，防止伪造教材来源"""
    cleaned, n = self._DEGRADED_CITATION_RE.subn("", text)
    if n > 0:
        logger.warning("guard.degraded_citation_stripped", count=n)
    return cleaned
```

这确保即使模型在降级路径中不慎输出了引用标记，也会被安全移除。

## 6. 记忆与 RAG 协同改进

### 6.1 RAG 查询改写读取滚动摘要

**改动文件**：`kernel/rag.py` `_rewrite_query` 方法（第 186-242 行）

**改动前**：查询改写仅读取最近 4 条对话历史（`history[-4:]`），无法利用跨消息窗口的上下文压缩信息。

**改动后**：新增读取 `conversations.summary`（滚动摘要），指代消解时拥有更完整的历史上下文。

```python
# 读取对话摘要（如果有）
summary_text = ""
if conversation_id and db:
    try:
        from app.models.conversation import Conversation
        conv_result = await db.execute(
            select(Conversation.summary).where(
                Conversation.id == conversation_id
            )
        )
        summary_text = conv_result.scalar() or ""
    except Exception:
        pass  # 摘要读取失败不影响改写

# 动态构建改写 prompt（含摘要上下文）
prompt_parts = ["请将以下多轮对话中的最新问题改写为一个独立的、完整的问题。"]
# ...
if summary_text:
    prompt_parts.append(f"\n对话摘要：\n{summary_text[:500]}")
if history_text:
    prompt_parts.append(f"\n对话历史：\n{history_text}")
```

**效果**：当对话已超过 4 条消息时，滚动摘要包含了更早的讨论焦点，改写器能更准确地消解"它"、"这个"等指代。例如用户第 15 轮说"它的导数怎么求"，改写器结合摘要可知"它"指代第 3 轮讨论的某个函数。

### 6.2 摘要触发阈值下调

**改动文件**：`kernel/memory.py`

| 参数 | 修改前 | 修改后 | 说明 |
|------|-------|-------|------|
| `SUMMARY_TRIGGER_COUNT` | 12 | 8 | 每 8 条消息（4 轮对话）即触发摘要更新，更早压缩上下文 |
| `SUMMARY_MAX_CHARS` | 400 | 300 | 摘要更精简，为对话历史留更多 token 预算 |

```python
# 修改后
SUMMARY_TRIGGER_COUNT = 8   # 每 8 条消息（4 轮对话）触发
RECENT_MESSAGES_LIMIT = 12  # 最近消息窗口（不变）
SUMMARY_MAX_CHARS = 300     # 摘要最大长度
```

### 6.3 修复 prompt 与常量矛盾

**问题**：修改前 `SUMMARY_PROMPT` 中要求"不超过 300 字"，但 `SUMMARY_MAX_CHARS = 400`，两者矛盾，模型可能生成 400 字摘要但 prompt 要求 300 字，或反之。

**修复**：将 `SUMMARY_MAX_CHARS` 从 400 下调至 300，与 `SUMMARY_PROMPT` 中的"不超过 300 字"完全一致：

```python
SUMMARY_PROMPT = r"""你是一个数学对话摘要压缩器。请将以下对话历史压缩为不超过300字的摘要。
...
请输出压缩后的摘要（不超过300字）："""

SUMMARY_MAX_CHARS = 300  # ← 与 prompt 中"300字"一致
```

### 6.4 协同效果

```
对话第 1-4 轮（8 条消息）
    │
    ▼ 触发摘要更新（阈值 8）
滚动摘要生成（≤300 字）
    │
    ├──→ RAG 改写：读取摘要 + 最近 4 条历史 → 指代消解更准确
    │
    └──→ ContextAssembler：注入摘要 + 最近消息 → 降级路径上下文完整
```

摘要更早生成（4 轮 vs 6 轮）+ 改写读取摘要 + 降级路径注入摘要，三者协同确保多轮对话的上下文连续性。
